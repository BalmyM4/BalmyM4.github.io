Historia

De la estandarización de redes a Kerberos

Capa de aplicación

La capa de aplicación se entiende dentro de la historia del modelo OSI. En los años 70 y 80, los sistemas informáticos comenzaron a conectarse cada vez más, pero cada fabricante usaba soluciones propias de red.

Para resolver ese problema, ISO y el CCITT impulsaron un modelo estandarizado. En 1984, ISO publicó formalmente el modelo OSI, cuyo enfoque por capas ayudó a que sistemas diferentes pudieran comunicarse.

Dentro de ese modelo, la capa de aplicación es la que usan las aplicaciones de software para iniciar la comunicación con la red.

Kerberos

1967 · Bellfield

Usaba un Centro de Distribución de Claves para comunicaciones seguras.

1978 · Needham-Schroeder

Adaptó ideas previas para redes de computadoras.

1985 · MIT

Eric Jaeger y Cliff Neuman agregaron nuevas ideas al protocolo como parte de su tesis.

1987 · Kerberos

Cliff Neuman y Steve Miller desarrollaron Kerberos en el proyecto Athena del MIT.

Pregunta de refuerzo · Historia del modelo OSI

Selecciona el nivel de dificultad.

¿Qué problema impulsó la creación de un modelo de red estandarizado como OSI?

  1. Que todos los fabricantes usaban exactamente los mismos protocolos.
  2. Que las redes solo funcionaban si usaban Kerberos.
  3. Que cada fabricante desarrollaba sus propias soluciones de red.
  4. Que las aplicaciones ya no necesitaban comunicarse en red.

¿Qué ventaja aportó organizar el modelo OSI en capas?

  1. Permitió que cada capa resolviera una función específica dentro de la comunicación.
  2. Permitió que cada fabricante mantuviera protocolos incompatibles sin afectar la red.
  3. Hizo que la capa de aplicación se encargara de todas las funciones del modelo.
  4. Hizo que las aplicaciones se comunicaran sin depender de servicios de red.

Si una empresa usaba una arquitectura propia antes de la estandarización OSI, ¿cuál sería el problema más probable al conectarse con sistemas de otro fabricante?

  1. Podrían comunicarse solo si ambos sistemas usaban la misma aplicación visible.
  2. Podrían tener dificultades por diferencias en arquitectura o protocolos subyacentes.
  3. Podrían conectarse automáticamente porque OSI ya funcionaba como estándar global.
  4. Podrían evitar la capa de aplicación y comunicarse directamente con Kerberos.

Pregunta de refuerzo · Historia de Kerberos

Selecciona el nivel de dificultad.

¿En qué proyecto se desarrolló Kerberos como sistema de autenticación seguro?

  1. Proyecto Bellfield.
  2. Proyecto Athena del MIT.
  3. Proyecto CCITT.
  4. Proyecto OSI de ISO.

¿Qué relación tiene el protocolo Needham-Schroeder con Kerberos?

  1. Fue una evolución posterior de Kerberos.
  2. Fue creado por ISO para definir la capa de aplicación.
  3. Fue un antecedente que influyó en el desarrollo de Kerberos.
  4. Fue el primer nombre oficial del modelo OSI.

Considerando Bellfield, Needham-Schroeder y Kerberos, ¿qué idea técnica se conserva en la evolución de estos sistemas?

  1. La autenticación depende de enviar la contraseña directamente al servicio final.
  2. La seguridad se apoya en un centro confiable que participa en la distribución de claves o tickets.
  3. El cliente obtiene acceso sin que exista una entidad que valide su identidad.
  4. El servidor final sustituye al KDC para evitar depender de terceros confiables.

Objetivos

Para qué sirven

Capa de aplicación

Su objetivo es proveer servicios de red a las aplicaciones usadas por los usuarios, permitiendo comunicación eficaz y segura entre programas dentro de una red.

Kerberos

Su objetivo es ofrecer autenticación segura para que usuarios verifiquen su identidad sin enviar contraseñas por la red, reduciendo el riesgo de intercepción.

Pregunta de refuerzo · Objetivos

Selecciona el nivel de dificultad.

¿Cuál es el objetivo principal de la capa de aplicación?

  1. Cifrar todos los datos antes de que salgan del equipo.
  2. Proveer servicios de red a las aplicaciones usadas por los usuarios.
  3. Emitir tickets para que los usuarios accedan a servicios.
  4. Verificar que los relojes de los equipos estén sincronizados.

¿Qué diferencia mejor el objetivo de la capa de aplicación frente al objetivo de Kerberos?

  1. La capa de aplicación facilita servicios de red para programas; Kerberos verifica identidades sin enviar contraseñas por la red.
  2. La capa de aplicación autentica usuarios mediante tickets; Kerberos permite que las apps usen servicios como web o correo.
  3. La capa de aplicación sustituye los servicios de red; Kerberos solo organiza cómo se muestran los datos.
  4. La capa de aplicación y Kerberos cumplen el mismo objetivo, pero en capas distintas del modelo OSI.

Una empresa quiere que sus usuarios accedan a correo, archivos y plataformas internas, pero sin escribir ni enviar su contraseña cada vez que usan un servicio. ¿Qué análisis es más correcto?

  1. La capa de aplicación elimina la autenticación porque ya proporciona los servicios de red.
  2. Kerberos reemplaza los servicios de red, por eso el correo y los archivos ya no dependen de aplicaciones.
  3. Kerberos permite autenticar el acceso a servicios sin reenviar la contraseña continuamente.
  4. La capa de aplicación cifra las contraseñas y el KDC solo decide cómo se representan los datos.

Capa de aplicación

Características principales

Interfaz entre usuario y red

Permite que programas y aplicaciones accedan a recursos de red para enviar y recibir información.

Servicios de red

Proporciona servicios que las aplicaciones necesitan para funcionar correctamente dentro de una red.

Protocolos de aplicación

Usa protocolos como HTTP, HTTPS, FTP, SMTP, POP3, IMAP y DNS.

Recursos compartidos

Permite acceder a archivos, impresoras, servidores, bases de datos y otros recursos disponibles en red.

Comunicación entre aplicaciones

Facilita el intercambio de información entre programas, incluso en dispositivos o sistemas distintos.

Autenticación y acceso

Algunas aplicaciones verifican la identidad del usuario antes de permitir el acceso a servicios o recursos.

Ejemplos de servicios

Navegación web, videollamadas, streaming, almacenamiento en la nube y plataformas educativas.

Protocolos clave

HTTP/HTTPS, FTP, SMTP, POP3, IMAP y DNS permiten diferentes servicios de comunicación en red.

Acceso a recursos

La capa permite acceder a archivos, impresoras, servidores, bases de datos y otros recursos compartidos.

Pregunta de refuerzo · Características de aplicación

Selecciona el nivel de dificultad.

¿Qué función cumple la capa de aplicación como interfaz entre usuario y red?

  1. Permite que los programas accedan a servicios y recursos de red.
  2. Convierte todos los datos en señales físicas para enviarlos.
  3. Asigna direcciones IP a cada equipo conectado.
  4. Divide los paquetes antes de enviarlos por la red.

¿Cuál opción describe mejor una característica general de la capa de aplicación?

  1. Se limita a mostrar datos ya procesados por otras capas.
  2. Permite que diferentes programas intercambien información mediante servicios de red.
  3. Reemplaza los protocolos de comunicación cuando las aplicaciones son compatibles.
  4. Permite que las aplicaciones accedan a servicios, pero sin intervenir en la comunicación entre programas.

Dos aplicaciones en equipos distintos necesitan intercambiar información dentro de una red. Además, el servicio debe funcionar aunque los equipos usen sistemas operativos diferentes. ¿Qué análisis corresponde mejor a la capa de aplicación?

  1. La capa de aplicación permite el intercambio de información mediante servicios de red, aunque los sistemas sean distintos.
  2. La capa de aplicación solo funciona si ambos equipos usan el mismo sistema operativo y la misma aplicación.
  3. La capa de aplicación evita el uso de protocolos cuando las aplicaciones ya son compatibles.
  4. La capa de aplicación se limita a validar la identidad del usuario antes de permitir cualquier comunicación.

Pregunta de refuerzo · Protocolos y recursos

Selecciona el nivel de dificultad.

¿Cuál de los siguientes protocolos se relaciona con la navegación web?

  1. SMTP
  2. FTP
  3. HTTP/HTTPS
  4. IMAP

¿Qué relación existe entre los protocolos de aplicación y los servicios de red?

  1. Los protocolos de aplicación permiten que servicios como web, correo o transferencia de archivos funcionen.
  2. Los protocolos de aplicación sustituyen a los recursos compartidos dentro de una red.
  3. Los protocolos de aplicación solo se usan cuando no hay autenticación de usuarios.

Un usuario accede a una carpeta compartida, consulta una base de datos y usa correo institucional desde una misma red. ¿Qué idea explica mejor el papel de la capa de aplicación?

  1. Todos esos servicios dependen del mismo protocolo porque pertenecen a la misma capa.
  2. La capa de aplicación permite acceder a distintos recursos, pero cada servicio puede usar protocolos diferentes.
  3. La capa de aplicación solo interviene en el correo, porque los archivos y bases de datos pertenecen a hardware.
  4. Los recursos compartidos dejan de usar red cuando el usuario ya está autenticado.

Dónde y cuándo se emplea

Uso general

La capa de aplicación se emplea en sistemas y servicios de red donde los usuarios interactúan con aplicaciones que usan Internet o una red local. Es la capa más cercana al usuario porque permite la comunicación entre las aplicaciones y la red.

Funciona cada vez que una aplicación necesita comunicarse por red para ofrecer un servicio al usuario.

Características principales

Lo esencial de Kerberos

Autenticación mutua

Cliente y servidor verifican sus identidades.

TGT

Ticket inicial con validez limitada para pedir acceso a otros servicios.

Single Sign-On

El usuario inicia sesión una vez y accede a servicios permitidos.

Sin contraseña en red

La contraseña no viaja directamente por la red.

Marcas de tiempo

Ayudan a evitar ataques de repetición.

KDC

Centro de Distribución de Claves que actúa como tercero confiable.

Idea para exponer

Kerberos no “confía porque sí”; usa tickets emitidos por un servidor confiable.

Detalle importante

Requiere relojes sincronizados para que los tickets temporales funcionen correctamente.

Pregunta de refuerzo · Características

Selecciona el nivel de dificultad.

¿Qué significa que Kerberos use autenticación mutua?

  1. Que el cliente y el servidor verifican sus identidades.
  2. Que el usuario escribe su contraseña en cada servicio.
  3. Que todos los tickets tienen duración ilimitada.
  4. Que el KDC deja de participar después del inicio de sesión.

¿Qué relación hay entre el TGT y el Single Sign-On en Kerberos?

  1. El SSO reemplaza al TGT cuando el usuario entra al primer servicio.
  2. El TGT permite solicitar acceso a servicios sin volver a introducir credenciales.
  3. El TGT guarda la contraseña para enviarla a cada servidor solicitado.
  4. El SSO elimina la necesidad de tickets dentro de la red.

Un usuario inicia sesión una vez y después accede a correo, archivos e intranet sin volver a escribir su contraseña. ¿Qué interpretación es más precisa?

  1. El usuario ya no necesita autenticación porque el primer servicio validó toda la red.
  2. El KDC entrega la contraseña del usuario a cada servicio para evitar nuevos inicios de sesión.
  3. Kerberos usa tickets para permitir acceso a servicios autorizados después de la autenticación inicial.
  4. La autenticación mutua solo ocurre cuando el usuario cambia manualmente de contraseña.

Pregunta de refuerzo · Seguridad y tickets

Selecciona el nivel de dificultad.

¿Qué característica de Kerberos reduce el riesgo de que una contraseña sea interceptada en la red?

  1. Que los tickets tienen tiempo de vida.
  2. Que usa marcas de tiempo.
  3. Que la contraseña no viaja directamente por la red.
  4. Que el usuario puede acceder a varios servicios.

¿Para qué sirven las marcas de tiempo dentro de Kerberos?

  1. Para decidir qué aplicaciones pertenecen a la capa de aplicación.
  2. Para evitar que mensajes válidos sean reutilizados fuera de tiempo.
  3. Para guardar la contraseña cifrada dentro del ticket.
  4. Para sustituir al KDC cuando el servidor ya conoce al usuario.

Un atacante captura un ticket válido y trata de usarlo después. ¿Qué combinación explica mejor por qué Kerberos puede rechazar ese intento?

  1. El ticket tiene tiempo de vida y el autenticador incluye una marca temporal.
  2. El usuario cambia de aplicación y el servidor borra automáticamente todos los tickets.
  3. El KDC envía la contraseña al servidor para confirmar que el ticket sigue activo.
  4. El cliente evita al KDC y valida el ticket directamente con cualquier servicio.

Dónde y cuándo se emplea

Kerberos en servicios reales

¿Qué es el KDC?

El KDC es el tercero de confianza en Kerberos. Emite tickets que demuestran la identidad de los usuarios.

Cuando el usuario inicia sesión, recibe un TGT. Después, si quiere entrar a una aplicación, carpeta o impresora, el KDC emite un ticket específico para ese servicio.

¿Cuándo se usa?

Se utiliza cuando una organización necesita autenticación segura sin reenviar contraseñas constantemente. Es común en empresas, universidades, bancos, centros de datos y entornos mixtos Linux + Windows.

Flujo resumido

1

El usuario inicia sesión.

2

El KDC valida identidad.

3

Se entrega un TGT.

4

Se solicita ticket para un servicio.

5

El usuario accede sin repetir contraseña.

Pregunta de refuerzo · Uso de Kerberos

Selecciona el nivel de dificultad.

¿Cuál es la función principal del KDC dentro de Kerberos?

  1. Sustituir todos los servicios de red de una organización.
  2. Enviar la contraseña del usuario a cada servidor solicitado.
  3. Convertir cualquier aplicación en un servicio público.
  4. Emitir tickets que permiten demostrar la identidad del usuario.

En una organización, un usuario inicia sesión y después accede a correo institucional, una carpeta compartida y una impresora de red sin escribir su contraseña otra vez. ¿Qué papel cumple Kerberos en ese escenario?

  1. Permite que cada servicio reciba la contraseña del usuario solo una vez.
  2. Autentica al usuario mediante tickets específicos para los servicios solicitados.
  3. Hace que todos los servicios usen el mismo ticket sin distinguir permisos.
  4. Evita que el KDC participe después del primer inicio de sesión.

Un usuario inicia sesión, recibe un TGT y luego quiere abrir una carpeta compartida. ¿Qué interpretación describe mejor el flujo de autenticación?

  1. El TGT permite pedir al KDC un ticket específico para acceder a la carpeta.
  2. El TGT reemplaza al KDC y autoriza directamente cualquier servicio.
  3. La carpeta compartida recibe la contraseña del usuario desde la computadora.
  4. El KDC participa solo después de que la carpeta ya permitió el acceso.

Uso en México

Ámbitos donde puede encontrarse

1

Empresas

Organizaciones que usan Active Directory emplean Kerberos como parte de su autenticación interna.

2

Gobierno

Dependencias con Active Directory o sistemas UNIX/Linux pueden usarlo para acceso a servicios internos.

3

Educación

Universidades lo pueden usar para SSO, correo, plataformas académicas, bibliotecas virtuales o Wi-Fi seguro.

4

Banca

Puede intervenir en cajeros, TPV, sucursales y acceso de empleados a plataformas internas.

Forma simple de identificarlo: si una red interna depende de Microsoft Active Directory, Kerberos funciona por defecto como protocolo de autenticación. En gobierno, su uso no siempre está documentado públicamente por razones de seguridad operativa.

Datos curiosos

Capa de aplicación y Kerberos

Capa de aplicación

No es la aplicación

La app es lo que usa el usuario; la capa de aplicación permite que esa app use servicios de red para comunicarse.

Es una capa muy atacada

Muchos ataques aprovechan errores humanos o fallas en formularios, inicios de sesión, páginas web y APIs.

APIs y servicios

Cuando una app consulta saldo, mensajes o clima, normalmente pide datos a un servidor mediante una API sobre HTTP/HTTPS.

Aplicación vs. presentación

Aplicación define qué servicio se usa; presentación prepara cómo se representan, protegen o entienden los datos.

Kerberos

Pulsera de acceso

Primero te autenticas una vez y después usas tickets para demostrar que ya fuiste validado.

La contraseña no viaja

La contraseña no se manda directamente por la red; se usan claves y tickets para comprobar la identidad.

Tickets temporales

Los tickets tienen una validez limitada para evitar que alguien los use mucho tiempo después si logra robarlos.

Relojes sincronizados

Kerberos usa marcas de tiempo; si los relojes están muy desfasados, la autenticación puede fallar.

Un ataque de repetición ocurre cuando un atacante captura un mensaje válido y lo vuelve a enviar para intentar engañar al sistema.

Pregunta de refuerzo · Datos curiosos de aplicación

Selecciona el nivel de dificultad.

¿Cuál afirmación describe mejor la relación entre una app y la capa de aplicación?

  1. Son exactamente lo mismo.
  2. La capa solo protege la contraseña del usuario.
  3. La app la usa el usuario; la capa permite usar servicios de red.
  4. La app solo funciona si usa Kerberos.

¿Por qué la capa de aplicación suele ser un punto atractivo para ataques?

  1. Porque ahí se generan los tickets de acceso.
  2. Porque ahí se sincronizan los relojes.
  3. Porque ahí se validan los KDC.
  4. Porque ahí interactúa el usuario.

Si una aplicación móvil consulta datos mediante una API, ¿por qué eso se relaciona con la capa de aplicación?

  1. Porque traduce los datos para que el sistema los cifre antes de enviarlos.
  2. Porque administra los tickets que autorizan el acceso al servidor.
  3. Porque establece el formato interno en que se representan los datos.
  4. Porque define el servicio de comunicación usado por la app.

Pregunta de refuerzo · Datos curiosos de Kerberos

Selecciona el nivel de dificultad.

¿Por qué Kerberos puede compararse con una pulsera de acceso?

  1. Porque validas una vez y luego muestras tickets.
  2. Porque evita identificarse al inicio.
  3. Porque cada servicio recibe tu contraseña.
  4. Porque el usuario crea sus propios permisos.

Si un atacante roba un ticket válido, ¿qué característica de Kerberos reduce el daño?

  1. Que el usuario cambia de aplicación.
  2. Que HTTP cifra todos los tickets.
  3. Que el KDC deja de emitir tickets.
  4. Que los tickets tienen tiempo de vida.

Si Kerberos evita enviar la contraseña por la red, ¿por qué aún necesita un servidor confiable como el KDC?

  1. Porque el KDC reemplaza todos los servicios de la red.
  2. Porque el KDC guarda los códigos de cada usuario.
  3. Porque alguien debe emitir y validar tickets confiables.
  4. Porque los tickets eliminan la autenticación mutua.
